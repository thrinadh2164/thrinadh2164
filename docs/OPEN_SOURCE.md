# 「 🌐 Open Source & Community Contribution Guide 」

> *"Open source is the foundation of modern artificial intelligence and software engineering. We grow faster when we build in public and collaborate openly."*

---

## ⚡ 1. Philosophy & Licensing

I am a strong advocate for accessible, clean, and well-documented open-source software:
* **MIT License Default:** Unless otherwise specified, all modular developer tools, utility libraries, and AI templates in this portfolio are released under the open MIT License.
* **Build in Public:** Believing in transparency, I share system architectures, prompt engineering frameworks, and benchmarking logs openly with the developer community.

---

## 🤝 2. Pull Request (PR) Guidelines

If you are contributing to any of my public repositories (`My_Awards`, `My_Certifications`, or custom AI tools), please adhere to the following workflow:
1. **Fork & Branch:** Create a feature branch from `main` using descriptive naming (e.g., `feature/hybrid-rag-retrieval` or `fix/jwt-auth-leak`).
2. **Clean Commits:** Write clear, concise commit messages explaining *why* a change was made, not just what was changed.
3. **Linting & Formatting:** Ensure code adheres to standard styling rules (Prettier for JS/TS, Black/Flake8 for Python, Ktlint for Kotlin).
4. **Documentation:** Any architectural change or new environment variable must be documented in the repository's `README.md`.

---

## 📬 3. Proposing New Features or AI Models

Have an idea for a multi-agent framework integration or a new RAG optimization?
* Open an **Issue** with the `[FEATURE]` tag detailing the use case, expected latency impact, and proposed architecture.
* Reach out directly via the channels listed in [COLLAB.md](./COLLAB.md) for co-developing larger systems.
